# v2.2.x Changelog

## v2.2.14-beta.5 (2026-04-25)

### Dashboard lifecycle (ref-counted, passive)
- **Dashboard now starts on the first `omc cc` and shuts down with the last one** â€” `src/cli/utils/proxy-lifecycle.ts` gains `maybeStopDashboard(reason?)`, a synchronous teardown probe invoked from every `omc cc` exit path (`cc-session-unix.ts` inline cleanup, `cc.ts stop` after killing the detached proxy, both code paths of `cleanup()`). It reads `proxy-sessions.json` + `proxy-instances.json` through the same stale-entry filters the dashboard UI already uses; when BOTH are empty it calls `stopDaemon()` and logs `[lifecycle] dashboard torn down (<reason>; origin=<auto|unknown>)`. Stays sync so it survives signal handlers that immediately `process.exit()`
- **Manual dashboards are preserved via `dashboard.origin`** â€” `startDashboard()` now takes `{ origin: "auto" | "manual" }` and writes a sibling `dashboard.origin` marker next to `dashboard.pid`. `origin === "manual"` is sticky: a later implicit `ensureDashboard({ origin: "auto" })` will NOT downgrade it. `omc proxy dashboard` passes `origin: "manual"` through `startDashboard`; `ensureDashboard()` passes `origin: "auto"`. `maybeStopDashboard` bails immediately when `readDashboardOrigin() === "manual"`, so `omc proxy dashboard` + subsequent `omc cc` â†’ exit cc leaves the user-launched dashboard alive
- **`stopDaemon()` unlinks both files atomically** â€” both `dashboard.pid` and `dashboard.origin` are removed on clean stop or stale-PID cleanup, preventing leftover origin markers from shadowing a future auto start

### High-severity fixes
- **B1 HIGH â€” `pid: 0` registration is no longer dangerous** â€” on some Windows bun spawn paths `proxyChild.pid` can be `0` briefly, which used to land in `proxy-sessions.json` and later cause `process.kill(0, â€¦)` to target the calling process / process group. `cc-session-unix.ts` now guards both `registerProxySession` call sites with `typeof pid === "number" && pid > 0` (warns + skips when not). `src/proxy/registry.ts cleanupStaleEntries()` filters `Number.isInteger(e.pid) && e.pid > 0 && isPidAlive(e.pid)` so any stale `pid=0` entries are purged. `cc.ts stopCommand` refuses to kill when `target.pid <= 0` and prints a clear warning instead
- **B2 HIGH â€” `settings-merger.ts` never silently overwrites a corrupt `settings.json`** â€” previously `loadSettings()` returned `{}` on `JSON.parse` failure, which made the next `saveSettings()` clobber the user's Claude Code config. Now it copies the corrupt file to `settings.json.corrupt-<unix-ts>.bak`, logs the backup path, and throws a new `SettingsCorruptError`. Callers (`install`, `doctor`, hooks/MCP merge paths) surface the error instead of silently writing `{}`. The original corrupt file is left on disk untouched so the user can fix it by hand

### Medium / low fixes
- **B3 MED â€” `memory-awareness.ts` no longer prints bogus "Proxy not detected" warnings** â€” the old heuristic compared the port parsed from `ANTHROPIC_BASE_URL` to a hard-coded `9090` (which has never been the proxy's port), warning every session with a dynamic omc port. New logic: if `OMC_PROXY_CONTROL_PORT` is set (definitive omc signal) â†’ no warning; else when `ANTHROPIC_BASE_URL` hostname is loopback, only suppress the warning if the port is a legacy default (18910/18920) OR appears in `proxy-sessions.json` / `proxy-instances.json` via the new `loadRegistryPorts()` helper; only foreign remotes get the warning
- **B4 MED â€” preference-awareness cache key now includes `cwd`** â€” `preference-awareness.ts` previously keyed its injected-prompt cache on `quickHash(prompt)`, so two projects that happened to send the same prompt shared injected preferences. Key is now `quickHash(\`${cwd ?? ""}:::${prompt}\`)`
- **B5 MED â€” `instance-registry.ts` RMW ops are now lock-protected** â€” `registerInstance` / `deregisterInstance` / `heartbeatInstance` now run under a tiny advisory lock (`openSync(lockPath, "wx")`, 10Ă— retries with ~20ms backoff + jitter, stale-lock auto-recycle after 5s) so concurrent proxy instances can't clobber each other's writes. Falls back to lockless RMW if the lock directory is not writable (best-effort; still strictly better than the old unconditional clobber)
- **B6 LOW â€” `comment-checker.ts` regex `lastIndex` bug** â€” dropped the `/g` flag from every pattern in `COMMENT_PATTERNS`. Reusing a `/g`-flagged RegExp across `.test()` calls caches `lastIndex` in V8 and produces flaky alternating matches. Only `.test()` is used here, so `/i` alone is correct
- **B7 LOW â€” `PID_FILE` renamed to `DASHBOARD_PID_FILE`** â€” the constant in `src/cli/utils/paths.ts` and its usages in `src/proxy/daemon.ts` now consistently spell it `DASHBOARD_PID_FILE`, eliminating the name/semantic drift with the new `DASHBOARD_ORIGIN_FILE`
- **B8 LOW â€” deleted dead `routeViaProxy` / `routeViaClaude` / `isProxyAvailable` + hard-coded 18910/18911** â€” `src/mcp/server/manager.ts` previously exported four helpers that were never referenced anywhere in the live code path but still hard-coded the legacy fixed proxy ports. Removed; the real proxy routing lives in `src/proxy/` and uses per-session ports. The module now only ships `isDirectCallViable` + `updateStatusFile`

### Notes
- Next `omc cc` spawn will pick up the new lifecycle â€” the currently-running proxy daemon (pid 220176) was spawned before this rebuild. `oh-my-claude proxy dashboard` / `--stop` continue to work unchanged
- `stopDaemon()` on Windows is a `TerminateProcess` wrapper; if the port is still held by a zombie from an older daemon, `ensureDashboard()` self-heals on the next `omc cc` by overwriting the stale pid file

## v2.2.14-beta.4 (2026-04-25)

### Added
- **`proxy.failClosed` config flag** â€” new Zod field on `ProxyConfigSchema` (default `true`). When a switched-path request to the selected provider raises (auth, quota, upstream 5xx, network), the proxy now returns `HTTP 502 {type:"error", error:{type:"upstream_error", provider, model, message}}` instead of silently bridging the request to the native Claude subscription. Applies to both `src/proxy/handlers/switched.ts` (per-request upstream failures) and the outer routing catch in `src/proxy/handlers/index.ts` (directive / session-state errors). ### Added
