# v2.2.x Changelog

## v2.2.14-beta.8 (2026-04-25)

### Robustness batch ŌĆö 23 fixes (13 HIGH + 10 MEDIUM)

Landed follow-up of the post-beta.7 audit. All fixes group by cluster below; the underlying change is a single new utility (`src/shared/fs/file-lock.ts`) that the rest of the clusters lean on.

#### Cluster 1 ŌĆö Reusable lock + atomic-write util (foundation)

- **NEW `src/shared/fs/file-lock.ts`** ŌĆö central primitives for every subsequent concurrency fix. Exports `withFileLockSync` / `withFileLock` (advisory `openSync(lock, 'wx')` with retry + stale-lock recycle, mirrors the `withRegistryLock` pattern introduced in beta.5), `atomicWriteText` / `atomicWriteJson` (temp-file + `renameSync` with optional POSIX `chmod`), and `loadJsonOrBackup` + `JsonCorruptError` (read ŌåÆ `JSON.parse` ŌåÆ schema validate; on any failure copies to `${path}.corrupt-<unix>.bak` and throws a typed error instead of silently recreating). Everything in Cluster 2 / 5 / 6 uses this module ŌĆö no more per-caller lock or atomic-rename boilerplate

#### Cluster 2 ŌĆö State-file concurrency (HIGH + MED)

- **HIGH-2 ŌĆö `src/cli/installer/settings-merger.ts` RMW on `settings.json` + `~/.claude.json`** ŌĆö every `loadSettings ŌåÆ mutate ŌåÆ saveSettings` block (install/uninstall hooks, MCP sync, statusline install/uninstall, legacy MCP cleanup) now runs inside `withFileLockSync`; `saveSettings` switched from `writeFileSync` to `atomicWriteJson` so partial writes during concurrent `omc install` / `omc cc` spawns can no longer half-corrupt the Claude settings file
- **HIGH-3 ŌĆö `src/hooks/user-prompt-submit/preference-awareness.ts` cache** ŌĆö `setCache` now writes under `withFileLockSync` with `atomicWriteJson` (tmpdir JSON file, previously vulnerable to two parallel prompt-submits racing each other into a zero-byte write)
- **HIGH-4 ŌĆö `src/memory/hooks/session.ts` state loading** ŌĆö `loadState` replaced its silent `catch { return defaults }` with `loadJsonOrBackup` + `SessionStateSchema`. On corruption the file is backed up, the corrupt path is recorded in a module-scope `Set`, and subsequent `saveState` calls short-circuit for that path ŌĆö preventing the old data-loss pattern where a corrupt state file was silently overwritten with defaults
- **HIGH-5 ŌĆö `src/proxy/state/instance-registry.ts` self-healing RMW** ŌĆö split `readInstances` into `readInstancesLockFree()` (read-only, no writes) and `pruneInstances()` (writes, always inside `withRegistryLock`). `registerInstance` / `deregisterInstance` / `heartbeatInstance` now call `pruneInstances()` inside their existing lock scope instead of racing with each other via the old self-healing read
- **MED-4 ŌĆö `src/proxy/state/switch.ts` writes under lock** ŌĆö `writeSwitchState` now runs inside `withFileLockSync(getSwitchStateLockPath(), ŌĆ”)` + `atomicWriteJson`; concurrent `mcp__oh-my-claude__switch_model` calls from MCP + CLI + dashboard can no longer interleave
- **MED-5 ŌĆö `src/proxy/registry.ts` session registry writes** ŌĆö `registerProxySession` / `unregisterProxySession` / `cleanupStaleEntries` now wrap every RMW in `withFileLockSync`; the WSL2 dual-write to the Windows-side path also locks the Windows sibling
- **MED-7 ŌĆö `src/memory/timeline.ts` TIMELINE.md writes** ŌĆö `writeTimeline` switched from `writeFileSync` (which truncates for the write window) to `atomicWriteText`; readers never see a partial TIMELINE again during a regen
- **MED-8 ŌĆö `src/shared/preferences/store.ts` validated + locked** ŌĆö `readJsonStore` now routes through `loadJsonOrBackup` with a new `PreferenceJsonSchema`; `writeJsonStore` uses `atomicWriteJson({ mode: 0o600 })`; the public `create` / `update` / `delete` methods wrap their RMW cycles in `withFileLockSync`

#### Cluster 3 ŌĆö Daemon / dashboard PID lifecycle (HIGH)

- **HIGH-6 + HIGH-7 ŌĆö `src/proxy/daemon.ts` + `src/cli/utils/paths.ts`** ŌĆö introduced `DAEMON_PID_FILE` (`proxy-daemon.pid`) alongside the pre-existing `DASHBOARD_PID_FILE` so the full-proxy daemon no longer silently overwrites the dashboard-only PID. Both spawn paths now use `openSync(pidPath, 'wx')` for atomic PID claim with liveness re-check on `EEXIST` (stale ŌåÆ unlink + retry; live ŌåÆ bail). `isRunning`, `startDaemon`, `startDashboard`, `stopDaemon`, and `getPid` all accept a `kind: 'daemon' | 'dashboard'` (default `'dashboard'` for backward-compat callers), and internal call sites now target the correct file

#### Cluster 4 ŌĆö Install path polish (HIGH)

- **HIGH-1 ŌĆö `src/cli/commands/core/update.ts` dist marker** ŌĆö the path check was `join(globalPkgDir, 'dist', 'cli.js')`, which has *never* existed (the real artifact is `dist/cli/cli.js`; same bug class as the beta-channel.ts fix from beta.6). Extracted `DIST_MARKER_REL = "dist/cli/cli.js"` in `src/cli/utils/paths.ts`; `update.ts`, `beta-channel.ts`, and `scripts/prepare.cjs` now all agree on the single canonical path so the "is this install built?" probe is consistent across every code path that asks the question

#### Cluster 5 ŌĆö Memory correctness (HIGH)

- **HIGH-9 ŌĆö `src/memory/indexer.ts` + `src/mcp/memory/helpers.ts` wire up `cacheEmbedding`** ŌĆö prior to beta.8 `cacheEmbedding()` had zero call sites; the hybrid-search ranking always paid the embedding cost online on every `recall`. New `indexMemoryWithEmbeddings(filePath, scope, projectRoot, provider)` runs after `indexFile`, pulls the chunks that still lack vectors for `(provider, model)`, embeds them, caches the vectors, and flushes ŌĆö so the first search after a save is already hybrid-ready. Gated behind new config knob `memory.embedding.onWrite` (default `true`); `indexNewMemory` in `src/mcp/memory/helpers.ts` reads the knob + falls back to plain `indexFile` when no embedding provider is resolved
- **HIGH-10 ŌĆö `src/memory/search.ts` + `src/mcp/memory/index.ts` capability vs executed tier** ŌĆö introduced `SearchEnvelope { results, capabilityTier, executedTier }`. `searchHybrid` now counts `embeddingsAvailable`; when zero the `executedTier` collapses to `'fts5'` even though `capabilityTier === 'hybrid'`. `memory_recall` and `memory_status` tools now report both fields so users see whether hybrid was merely *available* vs whether it actually *ran*. `searchMemories` (the legacy shape) stays behavioural-identical and simply unwraps `envelope.results`
- **HIGH-11 ŌĆö `src/memory/store.ts` path-traversal guard** ŌĆö new `MemoryIdInvalidError` + `validateMemoryId` (length Ōēż 200, regex `^[A-Za-z0-9_\-.:]+$`, rejects `..`, `\0`, path separators). Applied in `getMemory` / `updateMemory` / `deleteMemory` and in the internal `id` construction path of `createMemory`, so no attacker-supplied ID can walk out of the memory root. Exported from `src/memory/index.ts`

#### Cluster 6 ŌĆö Provider client cache invalidation (HIGH)

- **HIGH-12 ŌĆö `src/shared/providers/router.ts` mtime-based invalidation** ŌĆö the exported `clearClientCache()` had zero callers; live edits to `oh-my-claude.json` / `auth.json` were never picked up until the next process. Added `maybeInvalidateClientCacheByMtime()` which stats both files, distinguishes "never observed" from "observed+changed" with a `hadPrev` guard so the very first probe of a newly-created file doesn't false-positive, and clears `clientCache` when any watched file's mtime changed. Called on every `getProviderClient` entry
- **HIGH-12 follow-up ŌĆö `src/shared/auth/store.ts` touch on save** ŌĆö `saveAuthStore` now calls `utimesSync(AUTH_PATH, now, now)` after writing so byte-identical re-saves still bump mtime and trigger the invalidation above. Same file exports a new shared `writeSecretFile(path, content)` helper (`writeFileSync` + `setFilePermissions`) so future call sites don't have to duplicate the `chmod 0600` dance

#### Cluster 7 ŌĆö Credential permissions + proxy hardening (HIGH + MED)

- **HIGH-13 ŌĆö credential writer permissions** ŌĆö `scripts/minimax-login.ts`, `scripts/kimi-login.ts`, `scripts/aliyun-login.ts` now write via the new `writeSecretFile` helper so every saved cookie jar lands with POSIX `0o600`. Unused `writeFileSync` imports were removed from `src/shared/auth/minimax.ts` and `src/shared/auth/kimi.ts`
- **MED-1 ŌĆö `src/proxy/converters/responses-stream.ts` `response.failed`** ŌĆö upstream failures (Codex responses API) used to be rewritten as a clean `message_delta(stop_reason=end_turn)` + `message_stop`, making the Anthropic SDK think the turn succeeded and silently dropping the user's prompt. Now emits a proper Anthropic-shaped `error` event with `{type, message}` carried through from the upstream error payload; `content_block_stop` is still emitted for any still-open blocks so the SDK's framing stays valid
- **MED-2 ŌĆö `src/proxy/handlers/index.ts` body-size enforcement** ŌĆö added `readBodyWithLimit(req)` which returns `413 Payload Too Large` based on `Content-Length` (when present) or accumulated stream bytes (when chunked). Configurable via new `proxy.maxBodyBytes` (default 4 MiB). `handleMessages` runs it before `JSON.parse` so a malicious client can't force the proxy to buffer hundreds of MB before the parse fails
- **MED-3 ŌĆö `src/proxy/response/stream.ts` + `src/proxy/routing/provider-forward.ts` abort propagation** ŌĆö new `buildUpstreamSignal(clientSignal, timeoutMs)` uses `AbortSignal.any([clientSignal, AbortSignal.timeout(ŌĆ”)])` (graceful fallback when `any` is unavailable). Both `forwardToUpstream` (Anthropic-format path) and `forwardToProvider` (OpenAI-format path) now pass this composite signal to `fetch`, so when Claude Code disconnects mid-stream the proxy cancels the upstream `fetch` and frees the TCP connection instead of burning the full timeout
- **MED-6 ŌĆö `src/shared/auth/token-manager.ts` OAuth refresh timeout** ŌĆö `refreshOpenAI` now passes `AbortSignal.timeout(30_000)` to the token-endpoint `fetch`. Deduped refresh waiters used to hang forever when the endpoint stalled; they now reject with a clear network-error message within 30s
- **MED-10 ŌĆö `src/proxy/sanitizers/deepseek.ts` `thinking.disabled` honour** ŌĆö previously an explicit `thinking: {type: 'disabled'}` could still get silently promoted back into thinking mode if the user was on `deepseek-v4-pro` or had passed `opts.effort`. Now checked first via `isThinkingExplicitlyDisabled(body)`; if true, the sanitizer takes `applyFastPath` immediately (strips `output_config.effort`, never injects effort) ŌĆö user intent wins
- **MED-9 ŌĆö `src/hooks/session-start/auto-rotate.ts` deadline + audit trim** ŌĆö `callNarrativeAI`'s hard-coded `AbortSignal.timeout(45_000)` could overshoot the hook's 20s wall-clock and delay session startup; now derives `effectiveTimeout = clamp(remaining-from-deadlineMs, 1_000, AI_CALL_TIMEOUT_MS)`. `deadlineMs` is plumbed through `rotateGroup ŌåÆ callNarrativeAI`. ## v2.2.14-beta.7 (2026-04-25)

### Critical install fix, continued ŌĆö ship prebuilt `dist/` on `dev`

- **Why another beta so soon** ŌĆö beta.6's `prepare`-on-install strategy solved the missing-`dist/` case in principle, but two real-world failure modes kept biting the same user: (1) `bun 1.3.13` silently skipped the `prepare` lifecycle on `bun add -g <gh-tarball>` (no scripts ran, just shim creation + dep install in <2s; matches bun's default trust policy that blocks install-lifecycle scripts), and (2) **bun's global `node_modules` is flat**, so when both `@lgcyaxi/oh-my-claude` (declares `zod: ^3.24.0`) and `oh-my-opencode` (declares `zod: ^4.3.0`) are installed together, oh-my-opencode's constraint wins and `zod@4.3.6` is what lands on disk. Even if `prepare` had run, `bun build` would then bundle zod 4 into the fallback build ŌĆö producing a mixed-major bundle that would break at runtime
- **Fix** ŌĆö `dev` branch now ships a prebuilt `dist/` inside the git tree (force-added past the root `.gitignore` via `git add -f dist/`). The GitHub tarball therefore contains a ready-to-run bundle with our locked `zod@3.25.76` inlined at build time. `bin/oh-my-claude.js` finds `dist/cli/cli.js` immediately, skips every fallback, and the runtime never re-resolves `zod` from bun's global `node_modules`. End result: `bun add -g @lgcyaxi/oh-my-claude@https://github.com/lgcyaxi/oh-my-claude/tarball/dev` works even when zod 4 is sitting in the same flat global tree
- **Release discipline** ŌĆö [CLAUDE.md](../../CLAUDE.md) Release Process now documents the new dev-branch flow: `bun run build:all` ŌåÆ `git add -f dist/` ŌåÆ commit with the version bump + changelog in one commit ŌåÆ push to `dev`. `main` stays source-only and relies on the `prepare` script to rebuild `dist/` at `npm publish` time; when squash-merging `dev` ŌåÆ `main` you must `git rm -r --cached dist/` to drop the shipped bundle from the stable branch history
- **beta.6's `prepare` + hardened bin are kept as belt-and-suspenders** ŌĆö no changes to `scripts/prepare.cjs` or `bin/oh-my-claude.js`. If `dist/` ever goes missing (e.g. linked-checkout dev workflow, or a user hand-deletes it), prepare still rebuilds it and the bin still prints an actionable error instead of silently spawning the TS source. The actionable error the user saw in the wild (`dist/cli/cli.js was not found` with install links) is exactly the behaviour we wanted ŌĆö this release removes the underlying cause so the error doesn't trigger on a fresh dev-channel install

### Root cause one-liner

Bun global node_modules is flat + `oh-my-opencode@^4.3.0 zod` beats our `^3.24.0 zod` ŌåÆ our TS source loaded against zod 4's file layout ŌåÆ bun parser trips on `zod/v3/types.js:1:1`. Shipping a prebuilt bundle with zod 3 inlined removes every runtime resolve of `'zod'` and makes the conflict irrelevant.

## v2.2.14-beta.6 (2026-04-25)

### Critical install fix (dev channel)
- **`bun add -g .../tarball/dev` now builds on install** ŌĆö the dev-channel GitHub tarball ships source only (`dist/` is gitignored), and `package.json` previously only had `prepack` (publish-time). Fresh installs on any machine without a pre-existing `dist/` fell through to `spawn("bun", ["run", "src/cli/cli.ts"])` in `bin/oh-my-claude.js`, which then failed with a cryptic `Unexpected ŌĆ” zod/v3/types.js:1:1` parse error under bun 1.3.13 on Windows. The bin now prints a precise actionable error (installed package dir, one-line `bun run build:all` fix, bun install links, suggestion to reinstall) and exits 1 instead of spawning the source path. Developers working from a linked checkout can opt in with `OMC_ALLOW_SOURCE_FALLBACK=1`
- **Replaced `prepack` with `prepare` in `package.json`** ŌĆö both run during `npm publish` / `npm pack`, but `prepare` additionally runs on git/tarball installs. ## v2.2.14-beta.5 (2026-04-25)

### Dashboard lifecycle (ref-counted, passive)
- **Dashboard now starts on the first `omc cc` and shuts down with the last one** ŌĆö `src/cli/utils/proxy-lifecycle.ts` gains `maybeStopDashboard(reason?)`, a synchronous teardown probe invoked from every `omc cc` exit path (`cc-session-unix.ts` inline cleanup, `cc.ts stop` after killing the detached proxy, both code paths of `cleanup()`). It reads `proxy-sessions.json` + `proxy-instances.json` through the same stale-entry filters the dashboard UI already uses; when BOTH are empty it calls `stopDaemon()` and logs `[lifecycle] dashboard torn down (<reason>; origin=<auto|unknown>)`. Stays sync so it survives signal handlers that immediately `process.exit()`
- **Manual dashboards are preserved via `dashboard.origin`** ŌĆö `startDashboard()` now takes `{ origin: "auto" | "manual" }` and writes a sibling `dashboard.origin` marker next to `dashboard.pid`. `origin === "manual"` is sticky: a later implicit `ensureDashboard({ origin: "auto" })` will NOT downgrade it. `omc proxy dashboard` passes `origin: "manual"` through `startDashboard`; `ensureDashboard()` passes `origin: "auto"`. `maybeStopDashboard` bails immediately when `readDashboardOrigin() === "manual"`, so `omc proxy dashboard` + subsequent `omc cc` ŌåÆ exit cc leaves the user-launched dashboard alive
- **`stopDaemon()` unlinks both files atomically** ŌĆö both `dashboard.pid` and `dashboard.origin` are removed on clean stop or stale-PID cleanup, preventing leftover origin markers from shadowing a future auto start

### High-severity fixes
- **B1 HIGH ŌĆö `pid: 0` registration is no longer dangerous** ŌĆö on some Windows bun spawn paths `proxyChild.pid` can be `0` briefly, which used to land in `proxy-sessions.json` and later cause `process.kill(0, ŌĆ”)` to target the calling process / process group. `cc-session-unix.ts` now guards both `registerProxySession` call sites with `typeof pid === "number" && pid > 0` (warns + skips when not). `src/proxy/registry.ts cleanupStaleEntries()` filters `Number.isInteger(e.pid) && e.pid > 0 && isPidAlive(e.pid)` so any stale `pid=0` entries are purged. `cc.ts stopCommand` refuses to kill when `target.pid <= 0` and prints a clear warning instead
- **B2 HIGH ŌĆö `settings-merger.ts` never silently overwrites a corrupt `settings.json`** ŌĆö previously `loadSettings()` returned `{}` on `JSON.parse` failure, which made the next `saveSettings()` clobber the user's Claude Code config. Now it copies the corrupt file to `settings.json.corrupt-<unix-ts>.bak`, logs the backup path, and throws a new `SettingsCorruptError`. Callers (`install`, `doctor`, hooks/MCP merge paths) surface the error instead of silently writing `{}`. The original corrupt file is left on disk untouched so the user can fix it by hand

### Medium / low fixes
- **B3 MED ŌĆö `memory-awareness.ts` no longer prints bogus "Proxy not detected" warnings** ŌĆö the old heuristic compared the port parsed from `ANTHROPIC_BASE_URL` to a hard-coded `9090` (which has never been the proxy's port), warning every session with a dynamic omc port. New logic: if `OMC_PROXY_CONTROL_PORT` is set (definitive omc signal) ŌåÆ no warning; else when `ANTHROPIC_BASE_URL` hostname is loopback, only suppress the warning if the port is a legacy default (18910/18920) OR appears in `proxy-sessions.json` / `proxy-instances.json` via the new `loadRegistryPorts()` helper; only foreign remotes get the warning
- **B4 MED ŌĆö preference-awareness cache key now includes `cwd`** ŌĆö `preference-awareness.ts` previously keyed its injected-prompt cache on `quickHash(prompt)`, so two projects that happened to send the same prompt shared injected preferences. Key is now `quickHash(\`${cwd ?? ""}:::${prompt}\`)`
- **B5 MED ŌĆö `instance-registry.ts` RMW ops are now lock-protected** ŌĆö `registerInstance` / `deregisterInstance` / `heartbeatInstance` now run under a tiny advisory lock (`openSync(lockPath, "wx")`, 10├Ś retries with ~20ms backoff + jitter, stale-lock auto-recycle after 5s) so concurrent proxy instances can't clobber each other's writes. Falls back to lockless RMW if the lock directory is not writable (best-effort; still strictly better than the old unconditional clobber)
- **B6 LOW ŌĆö `comment-checker.ts` regex `lastIndex` bug** ŌĆö dropped the `/g` flag from every pattern in `COMMENT_PATTERNS`. Reusing a `/g`-flagged RegExp across `.test()` calls caches `lastIndex` in V8 and produces flaky alternating matches. Only `.test()` is used here, so `/i` alone is correct
- **B7 LOW ŌĆö `PID_FILE` renamed to `DASHBOARD_PID_FILE`** ŌĆö the constant in `src/cli/utils/paths.ts` and its usages in `src/proxy/daemon.ts` now consistently spell it `DASHBOARD_PID_FILE`, eliminating the name/semantic drift with the new `DASHBOARD_ORIGIN_FILE`
- **B8 LOW ŌĆö deleted dead `routeViaProxy` / `routeViaClaude` / `isProxyAvailable` + hard-coded 18910/18911** ŌĆö `src/mcp/server/manager.ts` previously exported four helpers that were never referenced anywhere in the live code path but still hard-coded the legacy fixed proxy ports. Removed; the real proxy routing lives in `src/proxy/` and uses per-session ports. The module now only ships `isDirectCallViable` + `updateStatusFile`

## v2.2.14-beta.4 (2026-04-25)

### Added
- **`proxy.failClosed` config flag** ŌĆö new Zod field on `ProxyConfigSchema` (default `true`). When a switched-path request to the selected provider raises (auth, quota, upstream 5xx, network), the proxy now returns `HTTP 502 {type:"error", error:{type:"upstream_error", provider, model, message}}` instead of silently bridging the request to the native Claude subscription. Set `proxy.failClosed: false` in `~/.claude/oh-my-claude/config.json` to restore the legacy resilient fallback. Applies to both `src/proxy/handlers/switched.ts` (per-request upstream failures) and the outer routing catch in `src/proxy/handlers/index.ts` (directive / session-state errors). Rationale: silent fallback masked misconfigured API keys and billed the wrong account; users still get full coverage by opting into the old behaviour per-install
- **`qwen3.6-plus` vision tag** ŌĆö `models-registry.json` now carries `"note": "supports vision"` on the Aliyun Qwen 3.6 Plus entry; the qwen agent description was corrected to "Qwen 3.6 Plus (supports vision)"
