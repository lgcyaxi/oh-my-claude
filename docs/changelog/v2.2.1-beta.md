# v2.2.1-beta.12 - 2026-03-16

> Ollama Native Thinking Support, tmux Detection Fix on Linux

---

## Fixes

- **tmux Detection on Linux** — `detectTerminalBackend()` now uses `tmux -V` (installed check) instead of `tmux list-sessions` (requires running server). On a fresh terminal with no tmux server, `list-sessions` fails and Linux falls through to `'none'` (direct spawn, no multiplexer). `tmux -V` succeeds whenever tmux is installed, which is all that's needed since `omc cc` creates its own session.
- **Ollama Native Thinking Protocol** — Ollama supports Anthropic's thinking protocol natively. The proxy now keeps the `anthropic-beta` header for Ollama (previously stripped for all switched providers) and skips thinking-block sanitization, allowing proper extended thinking with Ollama-hosted models.
- **Thinking Content Forwarded to Dashboard** — `AnthropicCompatibleClient` now extracts thinking content separately and surfaces it in the `ChatCompletionResponse.thinking` field. The `/api/internal/complete` endpoint returns `thinking` alongside `content`, so the dashboard can display reasoning traces from thinking-capable models.

<details>
<summary>Technical Details</summary>

- `cc-routing-unix.ts`: `tmux list-sessions` → `tmux -V` in `detectTerminalBackend()`
- `provider-forward.ts`: conditional `anthropic-beta` header — kept for Ollama, stripped for others
- `sanitizers/index.ts`: new `ollama` case — only strips unsupported content types, preserves thinking
- `anthropic-client.ts`: separate `thinking` extraction, added to response type
- `types.ts`: `ChatCompletionResponse.thinking` optional field
- `control/internal.ts`: forward `response.thinking` in `/api/internal/complete` response

</details>

# v2.2.1-beta.11 - 2026-03-16

> Ollama Thinking Model Support, AI Rename Fix, Dashboard Favicon

---

<details>

- `ai-rename.ts`: `max_tokens` 100 → 1024 for thinking model compatibility
- `control/index.ts`: new `/favicon.ico` route returning inline SVG with 1-hour cache

> Code Splitting for Proxy Control Modules + Auto-Update Check

---

## Features

- **Auto-Update Check** — Non-blocking update notification for CLI commands:
  - Two-phase design: sync banner from previous run's cache (zero latency) + detached child process fetches latest version for next run
  - Stable: checks npm registry for newer `@lgcyaxi/oh-my-claude` version
  - Beta: checks GitHub API for newer dev branch commit
  - Cache at `~/.claude/oh-my-claude/.update-check.json` with 24-hour interval
  - Suppressed for `update`, `install`, `uninstall` commands and `NO_UPDATE_CHECK=1` env var
  - Banner shown before command output (e.g., "Update available: 2.2.1-beta.8 → 2.3.0 / Run: omc update")

## Refactoring

- **Code Splitting: sessions.ts → sessions/ subfolder** — Split the 1,084-line monolithic `src/proxy/control/sessions.ts` into 6 focused modules:
  - `sessions/types.ts` — Type definitions (SessionIndexEntry, SessionIndex, ConversationEntry, ContentBlock)
  - `sessions/path.ts` — Path utilities, directory scanning, project resolution
  - `sessions/parser.ts` — JSONL parsing (extractQuickMeta, parseConversation)
  - `sessions/ai-rename.ts` — AI-powered session rename endpoint
  - `sessions/handler.ts` — Route dispatcher and CRUD handlers
  - `sessions/index.ts` — Barrel re-export

- **Code Splitting: memory.ts → memory/ subfolder** — Split the 1,374-line monolithic `src/proxy/control/memory.ts` into 8 focused modules:
  - `memory/types.ts` — Type definitions (MemoryEntry, MemoryEntryWithPath)
  - `memory/path.ts` — Path resolution, project scanning
  - `memory/io.ts` — Frontmatter parsing, file reading
  - `memory/query.ts` — Cross-scope memory collection
  - `memory/timeline.ts` — Lightweight timeline regenerator
  - `memory/ai-ops.ts` — AI operations (daily, compact, clear, summarize)
  - `memory/handler.ts` — Route dispatcher, CRUD, operation router
  - `memory/index.ts` — Barrel re-export

<details>
<summary>Technical Details</summary>

- 14 new modules created: 6 in `src/proxy/control/sessions/`, 8 in `src/proxy/control/memory/`
- Sessions module: 1,084 lines → 6 focused modules (~180 lines average)
- Memory module: 1,374 lines → 8 focused modules (~170 lines average)
