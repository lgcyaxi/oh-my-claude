## v2.1.3-beta.73 - 2026-03-11

### Windows Debug Mode WezTerm Fix

**`src/cli/commands/session/cc-terminals-win.ts`**:
- **New coordinator script pattern** (`buildDebugCoordinatorScript()`) — creates a `.cmd` script that runs INSIDE the WezTerm window to handle pane splitting
- The coordinator uses `wezterm cli split-pane` from within the process (guaranteed mux access) to create [CC 65% | Proxy 35%] layout
- Replaces the old external mux polling approach (`trySpawnHostPane` + `waitForMux`) which failed because `--always-new-process` isolated the mux
- **Simplified `spawnProxyInWeztermWindow()`** — removed complex 3-stage flow (try mux → bootstrap GUI → poll mux → attach pane)
- Now just: build scripts → `wezterm start --workspace ... -- cmd.exe /d /c call coordinator.cmd`
- Removed `--always-new-process` flag from `launchInWezterm` (was isolating the mux)
- Removed unused code: `trySpawnHostPane`, `waitForMux`, `sleep`, `WezTermListEntry`, `normalizePaneId`
- **Added pane focus** — `wezterm cli activate-pane` call after successful split so CC pane has focus
- **Tab activation fix** — `activate-pane` call at end of `launchDetachedSession` to switch to the CC+proxy tab after all console output is printed (fixes Phase 1 inside-WezTerm: proxy pane was created in a new tab but never focused)

**`src/cli/commands/session/cc-session-win.ts`**:
- Simplified Phase 2 in `launchDetachedDebugProxy()` — removed pane orchestration from outside WezTerm (coordinator handles it internally)
- Added `'window'` presentation type alongside existing `'pane'`
- Removed `splitProxyIntoWeztermPane` dependency from debug launch deps
- Added final `activate-pane` call after session output to ensure CC+proxy tab is visible when launched from inside WezTerm

**Test Coverage**:
- 18 tests pass across both terminal and session test files, 24 total across all CLI tests
